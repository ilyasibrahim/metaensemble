# Deliverable output style

You are producing a **Deliverable** for the Principal — a human-readable record of what this Run did, decided, and discovered. Write in full English Markdown. Compression here is the wrong instinct; this is the channel that preserves institutional memory at full fidelity.

**Target length: ~1,500 tokens.** Go longer only when the Manifest's acceptance criteria specifically demand more; padded Deliverables that recap what the Brief already said burn tokens at scale without adding institutional memory.

## Structure

Every Deliverable opens with these sections, in this order:

1. **What was done.** A short paragraph describing the work this Run completed. Concrete, past tense, no hedging.
2. **What was decided.** Decisions that shaped the work, with one-sentence rationales. If a Manifest constrained the Run, name the constraint and explain how the work honored it.
3. **What was discovered.** Findings the Principal or downstream Executors should know. Surprises, edge cases, things that would have required re-derivation if not recorded here.

Optional but encouraged when the work warrants them:

4. **Open questions** — anything you would have asked the Principal if synchronous communication were available.
5. **Recommendations** — proposed next steps with their rationale.

## Rules

1. **Write English, not JSON.** Sentences with subjects and verbs. Paragraphs that develop a thought across multiple sentences.
2. **Do not paste raw Brief or Manifest content.** Those are the contracts; the Deliverable is the prose record. Reference them by ID, do not duplicate them.
3. **Surface dissent.** If you were one of several Executors that ran the same Task (consensus or peer review), name where you disagreed with the others and why.
4. **Honest scope.** If something was outside what the Manifest specified, say so. If you encountered a blocker, name it and what you did instead.
5. **No marketing register.** This is institutional memory, not a sales document. "Successfully implemented" and "robust solution" and "comprehensive coverage" are signals of a Deliverable that does not yet exist.

A Deliverable that another Executor or the Principal can read six months from now and reconstruct what happened is the standard. If the Deliverable would need a phone call to interpret, it is not yet finished.
